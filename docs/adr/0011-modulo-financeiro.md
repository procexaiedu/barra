# Módulo Financeiro

Hoje o painel mostra um **bloco** financeiro no Dashboard (`dashboard/routes.py:_financeiro_bloco`) e fragmentos no Painel Geral (`painel/routes.py:179-217`), mas não há tela dedicada para Fernando administrar fluxo de caixa: registrar despesas (manuais e recorrentes), marcar pagamentos de repasse à modelo, e ver consolidado (bruto, repasses, despesas, líquido). Este ADR fecha o produto antes do código. Ver termo em `CONTEXT.md` ("Valor final", "Pix de deslocamento", "Lembrete de fechamento").

## Decisões

- **Receita = projeção de `atendimentos` fechados, sem tabela própria.** SELECT sobre `atendimentos WHERE estado='Fechado'`, com `valor_final` como bruto e `valor_final × percentual_repasse_snapshot/100` como repasse calculado. Single source of truth — mesmo padrão do **Mapa de clientes** (ADR 0008). Estorno usa `corrigir_registro` que já existe; auditoria via `barravips.eventos`. Zero código de sincronização. Receita "nasce" no `registrar_fechado` por consequência: o estado vira `Fechado` e o evento `fechado_registrado` é emitido — não há porta separada.
- **Data-âncora = `eventos.created_at WHERE tipo='fechado_registrado'`** (regime caixa contábil), mesma fonte do `_kpis` do dashboard. Alinhar é não-negociável: bloco financeiro do dashboard e tela do módulo precisam mostrar o **mesmo número** para a mesma janela. Competência via `bloqueios.inicio` é semanticamente mais correta mas exigiria migrar o dashboard também — adiado (sem ADR própria).
- **Pix de deslocamento fora do escopo P0.** Receita = só `valor_final` do programa. Pix de deslocamento (R$100 fixo, ver CONTEXT.md) é operacional: cliente paga ↔ agência chama Uber, saldo ≈ zero em média. Mantém o ADR 0004 limpo (`valor_final` não inclui Pix). Se virar relevante (margem do deslocamento), entra em P1 com categoria própria + rastreio de Uber.
- **Repasse pago em pagamentos livres**, tabela `financeiro_repasses_pagos` (sem vinculação 1:1 a atendimento). Linha = `(modelo, data, valor, forma, observacao, comprovante?)`. Espelha a operação real ("te pago R$5000 hoje pelo mês") em vez de ticar atendimento a atendimento. Saldo da modelo = `calculado − pago` por janela; pode ficar negativo após estorno, com badge vermelho e sem trava.
- **Atendimento sem `percentual_repasse_snapshot`** aparece como linha "pendente" destacada por modelo, com botão "Definir percentual retroativo" que faz UPDATE em massa no `atendimentos.percentual_repasse_snapshot` + emite `correcao_registro` por atendimento. CONTEXT.md já permite fechar sem snapshot — não advinhar com `modelos.percentual_repasse` atual (acordo pode ter mudado).
- **Categoria de despesa = enum fechado** `categoria_despesa_enum`: `anuncios | software | infraestrutura | juridico | taxas | deslocamento | pessoal | outro`. Mesmo padrão do projeto (`motivo_perda_enum`, `forma_pagamento_enum`, `perfil_fisico_enum` do ADR 0006). Sem hierarquia. Long tail vai em `outro`. Evoluir = `ALTER TYPE ADD VALUE`.
- **Despesa fixa = template + materialização sob demanda.** Tabela `financeiro_despesas_recorrentes` (template: categoria, valor, descricao, dia_do_mes, ativo_desde, inativo_em). Listagem de período faz UNION das pontuais com projeções dos templates ativos. Quando Fernando edita uma projeção (ex.: valor diferente do template em um mês), materializa em `financeiro_despesas` com `(recorrente_id, competencia_mes)` UNIQUE. Mudar valor do template afeta projeções futuras; histórico materializado fica imutável. Sem cron novo.
- **Sem rateio de despesa por modelo.** Despesa é da agência. Visão "por modelo" mostra só `receita_modelo / repasse_modelo / saldo_modelo`. Despesa global (anúncio Meta, software) entra apenas no líquido da agência. Modelo não "paga" Meta Ads no líquido dela.
- **Períodos**: `hoje | 7d | 30d | mes | tudo | custom` com default `mes`. `mes` é a unidade natural do financeiro (despesa fixa é mensal). BRT, comparativo período-anterior automático — reusa `_resolver_janela` do dashboard, estendido com case `mes` (1º do mês até hoje; anterior = mês anterior completo).
- **Forma de pagamento como dimensão.** Coluna no CSV + filtro opcional no top da Lista de Receitas. Dado já existe em `atendimentos.forma_pagamento`. Para `financeiro_repasses_pagos` reusa o `forma_pagamento_enum` mas restringe `cartao` via Pydantic (não muda enum SQL).
- **Exportação CSV linha-a-linha, 3 arquivos**, gerada no backend (`/export`). UTF-8-sig (com BOM), delimitador `;`. Filenames `receitas_<YYYY-MM>.csv` / `despesas_…` / `repasses_…`. Excel BR mostra acentos certos sem precisar configurar import.
- **RBAC herda guarda global** (`papel='fernando'` em `core/auth.py:103`). Sem código novo. Role `operador` não existe — `papel_usuario_enum` tem só `fernando` e `vendedor_read_only` (P1). Quando essa segunda aparecer, decide-se se vê financeiro num ADR separado.
- **Painel-only, IA por modelo bloqueada.** Mesmo padrão de ADR 0006 (Perfil físico) e ADR 0008 (Mapa): dado cross-modelo agregando todas as modelos = exposição para IA da modelo X violaria o isolamento por par cliente-modelo do `CONTEXT.md`. Endpoints sob `/v1/financeiro/*` são consumidos só pelo painel; `agente/` não importa `dominio/financeiro/`.
- **Sem backfill na migração.** Receita é projeção automática (atendimentos pré-existentes já contam). Despesas e pagamentos começam vazios. Fernando, se quiser zerar saldo histórico de uma modelo, lança 1 pagamento manual com `observacao='backfill histórico'`. Presumir "tudo já foi pago" no script automático seria errado.
- **Comprovante de pagamento opcional, MinIO bucket existente.** `comprovante_object_key text NULL` em `financeiro_repasses_pagos`. Upload via presigned PUT (reusa `core/storage.py`), download autenticado via presigned GET. Bucket `barra-media` com prefix `repasses/`. Sem auto-delete no P0; sem link público.
- **Correção pós-fechamento propaga automática.** Como receita é projeção, mudar `valor_final` via `corrigir_registro` muda o calculado em tempo real. Tabela `repasses_pagos` não muda. Saldo pode ficar negativo após estorno (badge vermelho + tooltip). Fernando compensa no próximo lançamento. Auditoria continua em `barravips.eventos`.

## Considered Options

- **Tabela `financeiro_receitas` denormalizada (INSERT no fechamento)**: permite estornos/ajustes manuais da receita independente do atendimento, mas exige sincronização com `corrigir_registro` e risco de divergência com o dashboard. Rejeitado — viola single source of truth.
- **Competência (`bloqueios.inicio`) como data-âncora em vez de caixa**: semanticamente mais correta ("rendimento do dia atendido"), mas exige migrar dashboard junto (caso contrário Fernando vê dois números). Rejeitada agora; pode virar ADR futura para mudar ambos.
- **Pix de deslocamento como receita separada + despesa Uber rastreada**: daria visibilidade do margem de deslocamento, mas exige campo novo no atendimento e tracking de Uber pago. Rejeitado P0; adiado.
- **Repasse com estado `pendente/pago/parcial` por atendimento**: granularidade que ninguém pediu; a operação paga em lote por modelo. Rejeitado.
- **Categorias em tabela CRUD ou texto livre**: CRUD vira tela extra com risco de proliferação ("Meta Ads" vs "Anuncio Meta"); texto livre fragmenta relatórios por ortografia. Rejeitados — enum fechado é o padrão do projeto.
- **Cron mensal materializa recorrentes no dia 1**: simples mas atualiza linhas pendentes ruins quando valor do template muda. Rejeitado a favor de materialização sob demanda.
- **Rateio de despesas global por faturamento**: complicação gratuita — Modelo X não "paga" Meta Ads que Fernando comprou. Rejeitado.
- **Guard `requer_admin` específico no financeiro** além do global: configurabilidade defensiva antes de existir outro papel. Rejeitado (CLAUDE.md §2).
- **Ganchos arquiteturais para Meta Ads/Google Ads já agora**: `origem` + `referencia_externa` antes de ter caso real. Rejeitado — quando aparecer decide-se com contexto.

## Consequences

- **Atendimentos pré-existentes sem `fechado_registrado`** ficam invisíveis no módulo. Antes de declarar M0, rodar `SELECT count(*) FROM atendimentos a WHERE estado='Fechado' AND NOT EXISTS (SELECT 1 FROM eventos e WHERE e.atendimento_id=a.id AND e.tipo='fechado_registrado')` em prod self-hosted. Se houver buracos significativos, fallback é `COALESCE(e.created_at, a.updated_at)`.
- **`_corrigir_registro` (`escaladas/service.py:199-245`) é estendido**: quando `novo_resultado='Fechado'` e estado anterior ≠ `'Fechado'`, emitir evento `fechado_registrado` adicional (além do `correcao_registro`). Sem isso, uma correção Perdido→Fechado deixa o atendimento sem porta de entrada no módulo. Mudança fora do módulo, no mesmo PR.
- **Bloco financeiro do dashboard fica como está.** Tela do módulo é a versão detalhada da mesma SQL base. Mudar uma sem a outra introduz divergência.
- **Idempotência da materialização recorrente** via índice único parcial `(recorrente_id, competencia_mes) WHERE recorrente_id IS NOT NULL` + CHECK amarrando os dois NULL juntos. `competencia_mes` sempre 1º do mês.
- **Migration aplicada à mão em prod self-hosted** (`make migrate` em prod aplicaria seeds — ver memória `migrations_pendentes_prod_selfhosted`).
- **MinIO bucket sem auto-delete**: aceitável no piloto; vira política de retenção em P1.
- **IA Admin (P1)** pode futuramente ler/escrever no financeiro por linguagem natural — Fernando perguntando "quanto a Modelo X recebeu este mês". A guarda atual (`papel='fernando'`) precisa virar ADR específica quando esse papel chegar.
