# Best-practices para colocar um agente LLM de WhatsApp em produção
### Síntese técnica para o atendente Elite Baby (LangGraph + Claude Sonnet 4.6 + prompt caching + STT/vision + persona compartilhada + isolamento por par cliente-modelo)

---

## Sumário executivo — as recomendações de maior impacto cross-eixo

1. **A fronteira de confiança real é a camada de dados/tools, não o system prompt.** Prompt injection não está resolvido nem no melhor modelo (Opus 4.5 ~1% ASR contra atacante Best-of-N; escala para ~63% em 100 tentativas em ambiente de coding via Gray Swan); injeção direta — o cliente digitando — é o pior caso de detecção. Imponha o isolamento por par cliente-modelo via filtro SQL em `repo.py` e least-privilege nas 5 tools, nunca pela boa-vontade do modelo em não citar dados de outra modelo. ([Anthropic Research — Prompt injection defenses](https://www.anthropic.com/research/prompt-injection-defenses); [Anthropic Engineering — How we contain Claude](https://www.anthropic.com/engineering/how-we-contain-claude)) — **crítico**

2. **Nunca combine `effort`/extended thinking com `tool_choice` forçado (`any`/`tool`) no mesmo nó** — a API retorna erro (reproduzido por LlamaIndex e Pydantic-AI). Audite os nós do graph que forçam tool para garantir que rodam sem thinking. ([Claude Docs — Define tools / Forcing tool use](https://platform.claude.com/docs/en/agents-and-tools/tool-use/define-tools)) — **crítico**

3. **`stop_reason` chega em 200 OK, não como exceção** — `refusal`, `max_tokens`, `model_context_window_exceeded` e `end_turn` vazio furam qualquer resiliência baseada só em `try/except` do SDK. Inspecione `response.stop_reason` ANTES de tratar o conteúdo, com ramos explícitos de controle (`Command/goto`). ([Anthropic Docs — Handling stop reasons](https://platform.claude.com/docs/en/api/handling-stop-reasons)) — **crítico**

4. **Ordene o prompt para o cache: tools → system (persona/FAQ/regras GERAIS) → messages (dados do par + turno).** Qualquer dado volátil (timestamp, dados do cliente, contador de turnos, imagem) antes do `system` derruba o hit rate a zero. Mantenha o breakpoint estritamente ANTES de qualquer dado isolado por par. ([Anthropic Docs — Prompt caching](https://platform.claude.com/docs/en/build-with-claude/prompt-caching)) — **crítico**

5. **Avalie o RESULTADO (estado final do banco), não o transcript, e use `pass^k` em fluxos críticos.** Confiabilidade decai exponencialmente (90% pass@1 → ~57% a k=8). Faça assert sobre efeitos observáveis (atendimento→Fechado com valor; bloqueio→concluido; `pix_status`; card em `envios_evolution`) e exija `pass^k` alto nos buckets de defesa/segurança. ([Anthropic Engineering — Demystifying evals](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents); [τ-bench, arXiv 2406.12045](https://arxiv.org/abs/2406.12045)) — **crítico**

6. **Persona drift é estrutural e PIOR em diálogos goal-oriented** (caso exato do agente de venda). Modelos maiores não drifam menos. Re-injete um lembrete de voz CURTO perto do fim do contexto (System Prompt Repetition + efeito de recência), fora do prefixo cacheado. ([arXiv:2402.10962](https://arxiv.org/html/2402.10962v1); [arXiv:2512.12775](https://arxiv.org/abs/2512.12775); [arXiv:2412.00804](https://arxiv.org/abs/2412.00804)) — **crítico**

7. **Mídia (STT/vision) é superfície de injeção indireta E risco de cache.** Mantenha STT/vision como steps separados que produzem TEXTO estruturado, injetado só em `messages` (depois do breakpoint); nunca deixe a imagem/áudio do cliente alterar o fluxo nem entrar no prefixo cacheado. Trate transcrição/extração como NÃO confiável (Whisper alucina ~40-100% em não-fala). ([Anthropic — Prompt caching](https://platform.claude.com/docs/en/build-with-claude/prompt-caching); [Anthropic — Vision](https://platform.claude.com/docs/en/build-with-claude/vision); [arXiv 2501.11378](https://arxiv.org/abs/2501.11378)) — **crítico**

8. **Instrumente cache hit rate e custo desde o piloto.** Leia `usage.cache_read_input_tokens`/`cache_creation_input_tokens` por turno; se ambos forem 0, o prefixo não foi cacheado (sem erro). Ligue tracing LangSmith com anonymizer de PII OBRIGATÓRIO (conteúdo íntimo). ([Anthropic — Prompt caching](https://platform.claude.com/docs/en/build-with-claude/prompt-caching); [LangChain — LangGraph Observability](https://docs.langchain.com/oss/python/langgraph/observability)) — **crítico**

## Engenharia de prompt e contexto

**[crítico] Ordene o prompt como tools → system → messages e mantenha o prefixo idêntico entre requisições.** O cache da Anthropic é um prefixo hierárquico: mudar `tools` invalida tools+system+messages; mudar `system` invalida system+messages; mudar `messages` só invalida messages. No agente, persona/FAQ/regras GERAIS (compartilhadas entre modelos) vão no `system`; dados do cliente isolados por par + turno vão em `messages`. Dado volátil antes do `system` zera o hit rate. ([Anthropic Docs — Prompt caching](https://platform.claude.com/docs/en/build-with-claude/prompt-caching), corroborado por Spring AI, MiniMax, mager.co)

**[alto] Instrumente `cache_read_input_tokens`/`cache_creation_input_tokens` para confirmar que o cache funciona.** Em Sonnet 4.6 o piso cacheável é 1.024 tokens (o system geral provavelmente já excede); blocos abaixo do piso não são cacheados e NENHUM erro é retornado — só detectável via `usage`. Se migrar para Opus 4.5+/Haiku 4.5, revalide contra o piso de 4.096. ([Anthropic Docs — Prompt caching, tabela de mínimos](https://platform.claude.com/docs/en/build-with-claude/prompt-caching), corroborado por MiniMax, AWS ML Blog)

**[alto] Use 2 breakpoints estáveis (fim das tool definitions, fim do system geral) e reserve os 2 restantes para a fronteira móvel da conversa.** Máximo 4 breakpoints/request; o sistema busca hits em até 20 blocos para trás de cada breakpoint, com o prefixo mais longo. Nunca ponha `cache_control` em conteúdo que muda toda requisição. ([Anthropic Docs — Prompt caching](https://platform.claude.com/docs/en/build-with-claude/prompt-caching), corroborado por Spring AI e DEV Community)

**[alto] Avalie TTL de 1h no prefixo estável (persona+FAQ+tools) por causa dos gaps longos do WhatsApp.** TTL padrão é 5min (refresh on hit); WhatsApp tem silêncios >5min entre turnos, então o cache de 5min expira e você paga re-escrita (1,25x) toda vez. O TTL de 1h paga 2,0x na escrita mas 0,1x em todas as leituras dentro de 1h. Ao misturar TTLs, ponha o bloco 1h ANTES do bloco 5m, ou a API retorna erro. ([Anthropic Docs — Prompt caching](https://platform.claude.com/docs/en/build-with-claude/prompt-caching); [Anthropic (X) — anúncio TTL 1h](https://x.com/AnthropicAI/status/1925633128174899453); [GitHub claude-code #46829](https://github.com/anthropics/claude-code/issues/46829))

**[alto] Não injete imagem do Pix nem áudio transcrito DENTRO do prefixo cacheado.** Imagens add/remove em qualquer ponto invalidam o cache de system+messages (tools sobrevive); `tool_choice` invalida só messages. Mantenha STT/vision como steps separados que produzem TEXTO, injetado após o último breakpoint, para que a presença de mídia não destrua o cache do prefixo persona+FAQ. ([Anthropic Docs — Prompt caching, tabela de invalidação](https://platform.claude.com/docs/en/build-with-claude/prompt-caching), corroborado por Spring AI)

**[alto] Posicione contexto longo no TOPO e a instrução acionável do turno por ÚLTIMO.** Para inputs grandes (20k+ tokens), colocar dados longos acima da query melhora a qualidade em até 30% em testes da Anthropic (especialmente multi-documento). Histórico/observações do par + dados da modelo primeiro; a pergunta acionável imediatamente antes da geração — casa com o caching (estável em cima, acionável embaixo) e com o viés de recência. ([Anthropic Docs — Long context tips](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/long-context-tips); [Prompting best practices](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices); corroborado por Simon Willison)

**[alto] Curadoria agressiva de contexto: trate cada token como custo de atenção.** Contexto é "attention budget" finito; precisão/raciocínio degradam com o enchimento ("context rot"). Busque o conjunto MÍNIMO de informação que delineia o comportamento esperado (mínimo ≠ curto). Não empilhe histórico ilimitado de WhatsApp; mantenha persona/FAQ/regras enxutas + dados relevantes do par; ponha o que mais importa no início e no fim, evitando enterrar regra crítica no meio. ([Anthropic Engineering — Effective context engineering](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)) ⚠️ **parcial/incerto:** o termo "lost-in-the-middle" não aparece neste artigo da Anthropic — é conceito da literatura (Liu et al. 2023, arXiv:2307.03172) frequentemente acoplado, mas não atribuível à Anthropic aqui. O núcleo (attention budget / context rot / conjunto mínimo) é sólido.

**[alto] Implemente compaction + note-taking externo para conversas longas.** A Anthropic recomenda sumarizar o histórico perto do limite preservando decisões/pendências e descartando outputs redundantes de ferramentas, e persistir estado em memória externa. Ao cruzar um limiar, gere resumo determinístico do par (recorrência, preferências, estado do atendimento) e descarte mensagens antigas; mantenha estado durável (atendimento, agenda, Pix) no Postgres, FORA da janela. ([Anthropic Engineering — Effective context engineering](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents); [Prompting best practices](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices))

**[médio] Use `trim_messages` por tokens como guarda-corpo + sumarização em nó dedicado.** ⚠️ **parcial/incerto:** `trim_messages` não trunca exclusivamente por tokens — aceita `token_counter`+`max_tokens` (modo token) ou contagem de mensagens; "por TOKENS" descreve o uso recomendado, não restrição da API. Prefira sumarização incremental (atualizar a cada turno) a re-sumarizar tudo, para cortar latência/custo. Cuidado: trocar mensagens antigas invalida o cache de messages — mantenha o ponto de corte estável quando possível. ([LangChain/LangGraph — Add memory](https://docs.langchain.com/oss/python/langgraph/add-memory); [trim_messages reference](https://reference.langchain.com/python/langchain-core/messages/utils/trim_messages); [Short-term memory](https://docs.langchain.com/oss/python/langchain/short-term-memory))

**[médio] No prompt geral, prefira heurísticas + 3-5 exemplos canônicos a árvores de if/else por cenário.** A "altitude" do prompt deve ser específica o bastante para guiar, flexível para dar heurísticas fortes; exemplos diversos e canônicos valem mais que listas exaustivas de edge cases. Coloque exemplos de diálogo reais (de `docs/agente/conversas-reais`) em tags XML no prefixo ESTÁVEL cacheado, pois não mudam por cliente. ([Anthropic Engineering — Effective context engineering](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents); [Prompting best practices](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices))

**[médio] No vision do Pix e em mensagens longas, faça a extração retornar campos/citações estruturadas, não imagem crua.** Para tarefas com documentos longos, pedir ao Claude para CITAR trechos relevantes em tags antes da tarefa corta o ruído; estruture com tags `<document>/<document_content>/<source>`. O passo de vision deve devolver valor/chave/status estruturados ao agente conversacional, mantendo o contexto principal enxuto e cacheável. ([Anthropic Docs — Long context tips / Ground in quotes](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/long-context-tips))

**[médio] Pré-aqueça o prefixo geral no startup do worker; use caching automático (1 breakpoint no fim do estático) no turno simples.** Pre-warming via `max_tokens:0` escreve a entrada de cache antes do tráfego (elimina a latência do primeiro miss); o caching automático move o breakpoint conforme a conversa cresce. Re-aquecer só faz sentido com tráfego — sem requests, manter quente custa escrita repetida; pondere TTL 1h se o silêncio for longo. ([Anthropic Docs — Prompt caching / pre-warming](https://platform.claude.com/docs/en/build-with-claude/prompt-caching), corroborado por mager.co)

## Tool use

**[crítico] NUNCA combine extended thinking/`effort` com `tool_choice` forçado (`any`/`tool`) no mesmo request.** A API retorna erro porque o forced tool use suprime os content blocks antes da tool call, impedindo o bloco de thinking obrigatório; forçar tool também prefixa a mensagem do assistant, eliminando resposta em linguagem natural. Com thinking, só `auto`/`none` são aceitos. Se um nó precisa de raciocínio E garantia de tool, escolha: `auto` + instrução no user turn, ou desligue thinking nesse nó. ([Claude Docs — Define tools](https://platform.claude.com/docs/en/agents-and-tools/tool-use/define-tools); [llama_index #19641](https://github.com/run-llama/llama_index/issues/19641); [pydantic-ai #2425](https://github.com/pydantic/pydantic-ai/issues/2425))

**[crítico] Devolva erro recuperável como `tool_result` com `is_error:true` + conteúdo instrutivo; erro de infra como exceção.** O Claude incorpora o erro e tenta 2-3 vezes com correções antes de desistir; a doc recomenda mensagens como "Rate limit exceeded. Retry after 60 seconds." em vez de "failed". Separe: (1) recuperável pelo modelo (horário fora da Disponibilidade, Pix abaixo do esperado, parâmetro ambíguo) → string + `is_error`; (2) infra (Evolution 5xx, Postgres down, Redis) → exceção tratada fora do loop (retry/escalada). ([Claude Docs — Handle tool calls](https://platform.claude.com/docs/en/agents-and-tools/tool-use/handle-tool-calls))

**[crítico] NUNCA coloque dado sensível como enum/const/nome-de-campo/regex no schema das tools.** Com `strict:true` o `input_schema` é compilado em gramática e cacheado por até 24h SEPARADO do conteúdo das mensagens, SEM as proteções de PHI. A doc proíbe PHI em property names, enum, const e pattern. Nome/telefone/JID/RG/CPF/endereço/chave Pix vão no conteúdo das mensagens (protegido); use enums genéricos no schema (taxonomia fechada de `motivo_perda` é vocabulário, seguro; uma lista de telefones de modelos como enum vazaria). ([Claude Docs — Strict tool use / Data retention](https://platform.claude.com/docs/en/agents-and-tools/tool-use/strict-tool-use))

**[alto] Ative `strict:true` nas tools críticas, mas valide `minimum`/`maximum` em código no service.** `strict:true` garante tipos corretos e nome válido, exige `additionalProperties:false`, e NÃO suporta schemas recursivos nem `minimum/maximum/minLength/maxLength`. Em `criar_bloqueio`/`registrar_extracao`/`escalar` garante valor numérico (não "dois reais"); mas a guarda do Piso de desconto e do valor mínimo de Pix TEM que ser código no service, não o schema. ([Claude Docs — Strict tool use](https://platform.claude.com/docs/en/agents-and-tools/tool-use/strict-tool-use); [Structured outputs / JSON Schema limitations](https://platform.claude.com/docs/en/build-with-claude/structured-outputs))

**[alto] Mantenha o catálogo enxuto (≈5 tools) e evite tools-irmãs com nomes quase idênticos.** "More tools don't always lead to better outcomes"; os modos de falha comuns são seleção errada de tool e parâmetros incorretos. Prefira um parâmetro `action`/enum a múltiplas tools — uma tool de envio com tag (texto/midia/card) é melhor que 4 tools de envio. ([Anthropic Engineering — Writing tools for agents](https://www.anthropic.com/engineering/writing-tools-for-agents); [Claude Docs — Best practices](https://platform.claude.com/docs/en/agents-and-tools/tool-use/define-tools)) ⚠️ **parcial:** o exemplo "notification-send-user vs notification-send-channel" não consta no artigo atual; o núcleo (consolidar, evitar ambiguidade de nomes) é confirmado.

**[alto] Descrições de tool são "by far the most important factor" — inclua o "quando NÃO usar".** Mire ≥3-4 frases por tool: o que faz, quando usar e quando não, significado de cada parâmetro, caveats; nomes inequívocos. Revise as 5 descrições para incluir "escalar só quando abaixo do piso", "enviar_midia só após o cliente qualificar". Com prompt caching ativo, descrições ricas têm custo amortizado. ([Claude Docs — Best practices](https://platform.claude.com/docs/en/agents-and-tools/tool-use/define-tools); [Anthropic Engineering — Writing tools for agents](https://www.anthropic.com/engineering/writing-tools-for-agents))

**[alto] Adicione `input_examples` onde o formato derruba parâmetros (Valor final BR, datas/horários, payload de `criar_bloqueio`).** A Anthropic mediu acurácia de parameter handling subindo de 72% para 90%; custo ~20-50 tokens (simples) a ~100-200 (aninhado); exemplos inválidos retornam 400; não funciona em server tools. Mostre padrões mínimo/parcial/completo (ex.: "R$ 1.200,50"). ([Claude Docs — Providing tool use examples](https://platform.claude.com/docs/en/agents-and-tools/tool-use/define-tools); [Anthropic Engineering — Advanced tool use](https://www.anthropic.com/engineering/advanced-tool-use))

**[alto] Decida paralelismo por nó: `disable_parallel_tool_use=true` em ações irreversíveis/handoff; leituras idempotentes em paralelo.** Por padrão o Claude chama múltiplas tools em paralelo e as calls do mesmo turno são NÃO-ordenadas. `criar_bloqueio`/`escalar`/envio ao cliente devem rodar com paralelismo desligado (evita duas reservas/dois cards); consultar agenda/preços pode rodar em paralelo. Não assuma ordem entre tool calls no código de despacho. ([Claude Docs — Parallel tool use](https://platform.claude.com/docs/en/agents-and-tools/tool-use/parallel-tool-use)) ⚠️ **parcial:** o detalhe "token-efficient tool use embutido nos Claude 4" não consta nesta página — trate como não verificado.

**[alto] Modele retornos de tool com alto sinal e NUNCA traga dados de outra modelo.** Retorne identificadores semânticos/estáveis e só os campos necessários; respostas infladas desperdiçam contexto (o Claude Code limita a 25.000 tokens com paginação/truncamento). `consultar_agenda` retorna janelas livres formatadas, não dumps internos; trunque histórico longo com mensagem orientando o modelo. O retorno respeita o isolamento por par. ([Anthropic Engineering — Writing tools for agents](https://www.anthropic.com/engineering/writing-tools-for-agents); [Claude Docs — high-signal responses](https://platform.claude.com/docs/en/agents-and-tools/tool-use/define-tools))

**[médio] Mantenha `tool_choice` estável ao longo da conversa.** Mudar `tool_choice` invalida blocos de mensagem cacheados (tool definitions e system permanecem). Use `auto` como padrão; se um nó precisa forçar tool, isole-o e aceite o custo de reprocessamento ali, monitorando cache hit rate por nó. ([Claude Docs — Forcing tool use](https://platform.claude.com/docs/en/agents-and-tools/tool-use/define-tools))

**[médio] Catálogo pequeno (≈5 tools) → Tool Search NÃO se aplica; o ganho vem de descrições+input_examples+response shaping.** A Anthropic só recomenda Tool Search/programmatic calling com 10+ tools ou >10K tokens de definições. A regra de "3-5 sempre carregadas" valida manter tudo carregado. ([Anthropic Engineering — Advanced tool use](https://www.anthropic.com/engineering/advanced-tool-use))

**[médio] Trate o gate de evals como cobertura de tool use real (seleção, parâmetros, paralelo vs sequencial, casos de NÃO chamar tool).** O BFCL (executável, ICML 2025, avaliação por AST) mostra que function calling continua frágil e exige avaliação dedicada. Inclua parâmetro ambíguo (formatos BR) e abstenção (recusar fora da Disponibilidade). ([BFCL — PMLR v267](https://proceedings.mlr.press/v267/patil25a.html); [Berkeley Function Calling Leaderboard V4](https://gorilla.cs.berkeley.edu/leaderboard.html)) ⚠️ **parcial:** "2.000+ pares" e "~0,73 para o melhor modelo" não conferem — o topo da V4 atual é ~0,72-0,77 e varia por versão; trate esses números como não verificados, a tese se mantém.

## Consistência de persona/voz

**[crítico] Trate deriva de voz como risco esperado em conversa longa e instrumente por número de turno.** Persona drift é mensurável e aparece cedo: LLaMA2-chat-70B/GPT-3.5 abandonam significativamente a persona dentro de ~8 rodadas, atribuído a "attention decay" (contexto longo → menor peso nos tokens iniciais do system). WhatsApp passa de 8-10 trocas facilmente; não confie que o system definido uma vez segura a persona até o fim. ([arXiv:2402.10962](https://arxiv.org/html/2402.10962v1), corroborado por Harvard VCG e repo dos autores)

**[crítico] Re-injete um lembrete de voz CURTO no user turn (System Prompt Repetition), fora do prefixo cacheado.** O SPR — re-injetar o núcleo da persona antes do turno do usuário — é a mitigação mais acionável (não exige acesso a pesos). O sistema já injeta "contexto+reminder no user turn"; este é o lugar natural para "responda como [persona], tom X, frases curtas". Mantenha curto e fora do bloco cacheado. ([arXiv:2402.10962v3](https://arxiv.org/html/2402.10962v3); [repo oficial](https://github.com/likenneth/persona_drift)) ⚠️ **parcial:** o paper define SPR de forma PROBABILÍSTICA (injetar com probabilidade p antes de cada utterance), não deterministicamente a cada turno; split-softmax existe como alternativa mas exige acesso a decoding.

**[alto] Priorize reforço de voz nos turnos de negociação/coleta de dados — o pior caso de drift.** A fidelidade degrada PIOR em diálogos goal-oriented que casual (competição entre manter o personagem e perseguir o objetivo); refreshes mitigam mas não eliminam. O agente é exatamente goal-oriented (fechar venda, coletar Pix, agendar). Combine reforço com eval que detecte quebra de personagem em turnos tardios. ([arXiv:2512.12775](https://arxiv.org/abs/2512.12775), corroborado por versão HTML) ⚠️ **parcial:** os limiares "~50 turnos / 100+" não foram localizados textualmente; o paper fala em ">100 rodadas" e degradação progressiva.

**[alto] Não assuma que Sonnet 4.6 protege a voz por ser forte — pode ser pior.** Em estudo com nove LLMs, modelos maiores tiveram MAIOR identity drift, e "assigning a persona may not help to maintain identity". A defesa é engenharia de prompt + re-injeção + avaliação, não "o modelo é bom o bastante". ([arXiv:2412.00804](https://arxiv.org/abs/2412.00804), corroborado por HTML v2 e OpenReview) ⚠️ **contestado:** é UM estudo (Chung-Ang Univ.), sem replicação independente em larga escala da relação tamanho-vs-drift; a generalização "modelo forte não protege" depende deste corpo ainda restrito, mas a implicação prática (não confiar só no modelo) é prudente.

**[alto] No `persona.md`/`regras.md`, inclua pares cenário→resposta-modelo, não só descrição de voz.** A Anthropic recomenda explicitamente (1) system com papel + personalidade detalhada (traços, background, maneirismos) e (2) "preparar Claude para cenários possíveis" com situações comuns e respostas esperadas. Ex.: cliente pede desconto abaixo do piso, áudio agressivo, "você é robô?". ([Anthropic — Keep Claude in character](https://platform.claude.com/docs/en/test-and-evaluate/strengthen-guardrails/keep-claude-in-character))

**[alto] Ancore a voz com 3-6 exemplos reais de mensagens (few-shot de estilo) no system cacheado.** Exemplos controlam estilo, não só conteúdo: o LLM capta comprimento de frase, vocabulário, pontuação, estrutura e transições melhor que instruções abstratas; a Anthropic lista "constrain with examples". Cure trechos curtos de `docs/agente/conversas-reais` (frases curtas, gírias, emojis, ausência de formalidade). ([Anthropic — Keep Claude in character](https://platform.claude.com/docs/en/test-and-evaluate/strengthen-guardrails/keep-claude-in-character); [Latitude](https://latitude.so/blog/how-examples-improve-llm-style-consistency))

**[alto] Posicione o lembrete de voz PERTO DO FIM do contexto (efeito de recência).** A Anthropic mede +30% de qualidade com a query/instruções no fim — contrapartida prática do attention decay. O reminder de voz re-injetado deve ser a última instrução antes da geração, não só lá em cima no system. ([Anthropic — Long context tips](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/long-context-tips), corroborado por Anthropic news e Simon Willison)

**[alto] Adicione ao eval suite um juiz de ESTILO/voz em 3 eixos, calibrado contra humanos.** LLM-as-Judge avalia consistência de voz com Prompt-to-Line (aderência por mensagem), Line-to-Line (contradições entre turnos) e Q&A Consistency (estabilidade de traços); um juiz Llama-3.1-70B atingiu 80,03% de concordância com humanos (acima da inter-humana, 72,91%). Rode especialmente em turnos tardios; calibre o juiz e mitigue vieses de posição/verbosidade. ([arXiv:2511.00222](https://arxiv.org/abs/2511.00222), corroborado por HTML e OpenReview)

**[alto] A persona/voz/FAQ compartilhada vive no PREFIXO ESTÁVEL cacheado; lembrete + dados por par ficam DEPOIS do breakpoint.** Cache hit exige prefixo 100% idêntico; é o caso de uso ideal de caching. Tensão real: o SPR pede repetir a persona perto do fim, mas isso não pode ser o mesmo bloco cacheado — use um lembrete curto/derivado, não recole o system inteiro. ([Anthropic — Prompt caching](https://platform.claude.com/docs/en/build-with-claude/prompt-caching); [Anthropic cookbook — prompt_caching.ipynb](https://github.com/anthropics/anthropic-cookbook/blob/main/misc/prompt_caching.ipynb))

**[médio] Não planeje ancoragem de voz via prefill — NÃO é suportado em Sonnet 4.6.** Prefill da resposta do assistant não funciona em Sonnet 4.6 (nem Opus 4.6/4.7/4.8); a Anthropic recomenda structured outputs ou instruções no system. Ancore por system detalhado + few-shot de estilo + re-injeção no user turn. Se algum doc/código pressupõe prefill para tom, é caminho morto. ([Anthropic — Keep Claude in character](https://platform.claude.com/docs/en/test-and-evaluate/strengthen-guardrails/keep-claude-in-character))

**[médio] No juiz de voz, use rubrica explícita + 2-4 exemplos rotulados, não prompt abstrato.** Exemplos few-shot com nota aumentam a consistência de pontuação e o alinhamento com humanos; rubricas + exemplos são o fator central de confiabilidade. Mitigue verbosidade/posição. ([Comet](https://www.comet.com/site/blog/llm-as-a-judge/); [Evidently AI](https://www.evidentlyai.com/llm-guide/llm-as-a-judge)) ⚠️ **parcial:** sem paper primário que quantifique o ganho "significativo"; é boa prática de engenharia consensual entre vendors, não achado empírico medido.

**[médio] Re-injetar a persona perto do fim defende também contra PRECEDÊNCIA, não só attention decay.** O cliente pode empurrar o agente para fora da voz ("responde sério", "admite que é IA"); o reminder mais recente compete com as instruções implícitas do cliente. ([Instruction Hierarchy, arXiv:2404.13208](https://arxiv.org/html/2404.13208v1); [APXML — Multi-Turn Attacks](https://apxml.com/courses/intro-llm-red-teaming/chapter-3-core-red-teaming-techniques-llms/multi-turn-conversation-attacks)) ⚠️ **parcial:** a premissa "later instructions take precedence" é imprecisa — a hierarquia é system > user, então o cliente não deveria simplesmente sobrescrever a persona; o mecanismo dominante é decaimento de aderência + hierarquia, não recência pura. A recomendação se sustenta porque ataca o drift.

## Segurança e isolamento

**[crítico] Não trate o system prompt como barreira de segurança; imponha isolamento por par na camada de dados.** Prompt injection não está resolvido nem no melhor modelo (Opus 4.5 ~1% ASR contra Best-of-N de 100 tentativas; a Anthropic chama de "risco significativo"). Aplique least-privilege nas 5 tools (nada de tool genérica de leitura cross-modelo) e imponha o isolamento por par em `repo.py` (filtro SQL), nunca pela boa-vontade do modelo. ([Anthropic Research — Prompt injection defenses](https://www.anthropic.com/research/prompt-injection-defenses); [Anthropic Engineering — How we contain Claude](https://www.anthropic.com/engineering/how-we-contain-claude)) ⚠️ **parcial:** os números do Opus 4.7 (~0,1% single, ~5-6% após 100) vêm do benchmark Gray Swan (terceiro), não das fontes Anthropic citadas; conceito corroborado por VentureBeat e The Decoder.

**[crítico] O cliente do WhatsApp é o canal de injeção DIRETA — não confie em classificador; defenda por arquitetura.** "Quando o usuário está digitando a instrução, não há nada de anômalo para um classificador pegar". Taxas baixas COMPÕEM ao longo de múltiplas interações (Gray Swan: Opus 4.5 de 4,7% em 1 tentativa → 63% em 100). A IA nunca executa ação financeira/agenda irreversível só com base no texto do cliente: mantenha gates determinísticos (regex de fechado/perdido só de Fernando/modelo; Pix por pipeline de vision com schema, não por afirmação do cliente). Monitore picos de tentativas de manipulação por número. ([Anthropic Research — Prompt injection defenses](https://www.anthropic.com/research/prompt-injection-defenses); [VentureBeat](https://venturebeat.com/security/prompt-injection-measurable-security-metric-one-ai-developer-publishes-numbers))

**[crítico] Defesa de produção é multicamada na ordem: ambiente primeiro, depois modelo, depois conteúdo externo.** "Projete para containment na camada de ambiente primeiro, depois oriente comportamento na camada de modelo." Mapeie as 5 tools como superfície de egress (permissão mínima, escopada por par); chave Pix e credenciais Evolution/Anthropic fora do alcance do contexto (em settings/keystore, anexadas pela humanização, nunca no prompt); inspecione input do cliente/mídia ANTES de entrar no contexto. ([Anthropic Engineering — How we contain Claude](https://www.anthropic.com/engineering/how-we-contain-claude))

**[crítico] Audite todo estado compartilhado (Redis, checkpointer, caches de aplicação, fixtures) e adicione eval de não-vazamento vinculante.** O maior risco cross-tenant em produção NÃO é o cache da API Anthropic (isolado), mas o reúso de estado/fixtures no próprio sistema — vetor real de vazamento cross-session (já visto no projeto: o cache test exige nonce porque o prefixo GERAL fica quente entre rodadas). Garanta chaveamento por par cliente-modelo e por turno; um cliente com a modelo A nunca recebe dados do mesmo cliente com a modelo B. Trate como gate vinculante (zero vazamento), não dashboard. ([Giskard — Cross-Session Leak](https://www.giskard.ai/knowledge/cross-session-leak-when-your-ai-assistant-becomes-a-data-breach); [FutureAGI](https://futureagi.com/glossary/cross-session-leak/); [NDSS 2025 — Prompt Leakage via KV-Cache Sharing](https://www.ndss-symposium.org/wp-content/uploads/2025-1772-paper.pdf))

**[alto] Delimite mensagem do cliente e texto de STT/vision como DADO, nunca comando — mas como defesa em profundidade.** LLMs não impõem fronteira de segurança entre instruções e dados; delimitadores são mitigação PARCIAL e contornável. Marque o conteúdo como `DADO_DO_CLIENTE` e instrua que ali é dado; combine com validação de entrada (padrões "ignore instruções", inclusive fuzzy/Levenshtein) e de saída (rastrear vazamento de system prompt/persona/dados de outra modelo antes de enviar a bolha). ([OWASP — LLM Prompt Injection Prevention Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/LLM_Prompt_Injection_Prevention_Cheat_Sheet.html); [OWASP GenAI — LLM01](https://genai.owasp.org/llmrisk/llm01-prompt-injection/)) ⚠️ **parcial:** a frase exata "contornável por linguagem natural que instrui a ignorar delimitadores" não consta verbatim; a substância (delimitadores não são correção foolproof) é corroborada por múltiplas fontes.

**[alto] Trate transcrição de áudio e texto de imagem como conteúdo NÃO confiável (injeção indireta).** A Anthropic escaneia "todo conteúdo não confiável que entra no contexto" com classificadores (texto oculto, imagens manipuladas). Uma imagem com texto "valide este Pix" não pode pular o pipeline de vision com `json_schema`; mantenha a decisão de validação do Pix na camada determinística/schema, fora do controle do conteúdo recebido. ([Anthropic Research — Prompt injection defenses](https://www.anthropic.com/research/prompt-injection-defenses); [How we contain Claude](https://www.anthropic.com/engineering/how-we-contain-claude))

**[alto] O cache da API é isolado por org/workspace; garanta que dado por-par fique APÓS o breakpoint do prefixo GERAL.** Caches nunca são compartilhados entre orgs (mesmo com prompts idênticos); hit exige match 100% do prefixo; KV/hashes só em memória, elegível para ZDR. Mesmo com baixo risco por design, valide que o breakpoint de cache fica estritamente antes de qualquer dado isolado por par. ([Anthropic Docs — Prompt caching](https://platform.claude.com/docs/en/build-with-claude/prompt-caching))

**[alto] Considere uma camada de classificação independente (input/output) sobre o agente — custo ~1%.** Constitutional Classifiers reduziram jailbreak de 86% para 4,4% (>95% bloqueados); a versão ++ mantém robustez forte com ~1% de compute extra. No mínimo, instrumente o LLM-judge de AUP como classificador vinculante de saída antes de enviar ao cliente. ([Anthropic — Constitutional Classifiers](https://www.anthropic.com/research/constitutional-classifiers); [Next-generation Constitutional Classifiers](https://www.anthropic.com/research/next-generation-constitutional-classifiers)) ⚠️ **parcial:** o "~1% compute" é da versão ++ (o sistema original custava 23,7%); o número "405 pesquisadores" não confere (foram 183 ativos no protótipo / ~339 no desafio público); "nenhum jailbreak universal passou" vale para a janela de prototipagem, não absolutamente.

**[alto] Avalie com fixtures multi-turn/Crescendo e meça ASR sobre a conversa inteira, não por mensagem.** Crescendo escala de inputs benignos a conteúdo proibido com ASR alta (98% binário GPT-4, 100% Gemini-Pro no AdvBench) usando mensagens individualmente inócuas; defesas single-turn não cobrem. WhatsApp é intrinsecamente multi-turn; o enforcement de AUP deve olhar o estado/contexto acumulado. ([arXiv 2404.01833](https://arxiv.org/abs/2404.01833); [USENIX Security 25 — Russinovich et al.](https://www.usenix.org/conference/usenixsecurity25/presentation/russinovich))

**[alto] Use juiz de AUP de modelo DIFERENTE do agente, calibrado contra golden set humano da Elite Baby.** LLM-as-judge tem vieses sistemáticos (position, verbosity, self-preference, family); modelos de fronteira excederam 50% de erro em testes de viés; calibrado, GPT-4 concorda >80% com humanos (paridade humano-humano). Regex pega o determinístico; o juiz decide o ambíguo e escala o resto a Fernando. ([MT-Bench, arXiv 2306.05685](https://arxiv.org/abs/2306.05685); [LangChain — Calibrate LLM-as-Judge](https://www.langchain.com/articles/llm-as-a-judge); [arXiv 2510.12462](https://arxiv.org/html/2510.12462v1))

**[médio] Adote JailbreakBench como suíte externa padronizada no gate de segurança; nunca aceite prefill do conteúdo do cliente.** Os 100 comportamentos servem de corpus de ataque. A robustez do Claude não é absoluta: modelos Claude foram jailbreakados via transfer/prefilling com até 100% de sucesso. ([JailbreakBench, arXiv 2404.01318](https://arxiv.org/abs/2404.01318); [GitHub](https://github.com/JailbreakBench/jailbreakbench); [Simple Adaptive Attacks, arXiv 2404.02151](https://arxiv.org/html/2404.02151v3)) ⚠️ **contestado/ressalva de versão:** os modelos jailbreakados a 100% via prefilling são Claude 2.x–3.5 Sonnet, não Opus 4.5/4.7; a tese "robustez não é absoluta, depende do vetor" sobrevive e reforça a regra de nunca aceitar prefill do cliente.

## Evals e quality gates

**[crítico] Avalie o RESULTADO (estado final do banco), não o caminho/transcript.** "Grade what the agent produced, not the path it took" — restringir o caminho acopla a eval a escolhas incidentais do harness e pune soluções válidas. Faça assert sobre efeitos observáveis (atendimento→Fechado com `valor_final`; bloqueio→concluido; `pix_status` gravado; card em `envios_evolution`); reserve transcripts para calibrar/depurar graders, nunca como pass/fail. ([Anthropic Engineering — Demystifying evals](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents); [τ-bench, arXiv 2406.12045](https://arxiv.org/abs/2406.12045); [Sierra](https://sierra.ai/uk/blog/tau-bench-shaping-development-evaluation-agents))

**[crítico] Adote `pass^k` (não só `pass@1`) como métrica de gate em fluxos críticos.** Confiabilidade decai exponencialmente (`pass^k = p^k`); 90% pass@1 → ~57% a k=8; SOTA resolve <50% e é inconsistente (pass^8 <25% no varejo). Rode K runs por fixture e exija `pass^k` alto nos buckets de defesa/segurança (nunca revelar persona/AUP, nunca travar por Pix, registrar valor correto), tolerando `pass@1` onde "um sucesso basta". ([τ-bench, arXiv 2406.12045](https://arxiv.org/abs/2406.12045); [Anthropic — Demystifying evals](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents))

**[crítico] Institucionalize error analysis como atividade central que define QUAIS evals escrever.** Open coding manual de 100+ traces frescos (pare quando ~20 traces não gerarem categoria nova), axial coding agrupando, anotando só a PRIMEIRA falha upstream em cascatas. Leia traces reais do LangSmith (corpus vivo), converta cada categoria nova em fixture; use 20-50 falhas reais como semente do gate. ([Hamel Husain — LLM Evals FAQ](https://hamel.dev/blog/posts/evals-faq/); [Anthropic — Demystifying evals](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents))

**[alto] Mantenha o gate determinístico (sem juiz) para formato/estado, e reserve o LLM-judge para o subjetivo.** Estratifique: determinístico para formato/estado, heurística para qualidade, LLM-judge para nuance, humanos para calibração; rode em cada PR e bloqueie deploy abaixo do threshold. Cubra SEM juiz: regex `finalizado/fechado [valor]`, transições (`Aguardando_confirmacao`→`Em_execucao` na foto de portaria; timeout 45min→`Perdido/sumiu`), idempotência de card por `card_message_id`, nunca-vazamento por regex. ([Anthropic — Demystifying evals](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents); [Hamel Husain — LLM Evals FAQ](https://hamel.dev/blog/posts/evals-faq/))

**[alto] Calibre o juiz LLM contra um held-out humano antes de torná-lo blocker (TPR/TNR + kappa).** GPT-4 atinge ~85% de concordância com humanos no MT-Bench (acima dos 81% humano-humano), mas isso exige medir concordância e mirar TPR/TNR altos. Só vincule se a divergência for baixa; re-calibre ao trocar prompt da persona ou versão do modelo. ([Anthropic — Demystifying evals](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents); [MT-Bench, arXiv 2306.05685](https://arxiv.org/html/2306.05685v4))

**[alto] Para o juiz de conformidade/persona, use rubrica + CoT + resposta de referência por fixture.** Reference-guided reduz falha de matemática de 70% para 15%; few-shot eleva consistência do GPT-4 de 65% para 77,5%; no estudo 2026, CoT dá +7,2pp e a combinada (swap+CoT+rubrica, 2 chamadas) +11,2pp para Claude Sonnet 4 (pico 70,0%, kappa 0,530). Como o projeto roda Sonnet como juiz, aceite o custo de 2 chamadas só no gate, não no nightly barato. ([MT-Bench, arXiv 2306.05685](https://arxiv.org/html/2306.05685v4); [Judging the Judges, arXiv 2604.23178](https://arxiv.org/html/2604.23178))

**[alto] Aplique swap de posição com regra de dupla-vitória em qualquer juiz pareado (A vs B).** No MT-Bench, ao trocar a ordem só o GPT-4 é consistente em >60% (GPT-4 65,0%; GPT-3.5 46,2%; Claude-v1 23,8%). Para juiz pass/fail absoluto o viés de posição é menos relevante — prefira esse formato no gate determinístico de comportamento. ([MT-Bench, arXiv 2306.05685](https://arxiv.org/html/2306.05685v4)) ⚠️ **ressalva de versão:** em juízes modernos o viés de posição ficou negligenciável (≤0,04); a magnitude histórica é real mas o swap rende pouco hoje (+0,2pp p/ Sonnet 4).

**[alto] No juiz de persona, neutralize estilo e não normalize por comprimento cegamente.** No "repetitive list attack" do MT-Bench, Claude-v1/GPT-3.5 falham em 91,3% (preferem a resposta inflada). O estudo 2026 reframa: o viés dominante hoje é de ESTILO/markdown (0,76–0,92), com posição negligível; em pares controlados há até preferência por concisão (Sonnet 4 a -0,76). Peça avaliação de conteúdo/conformidade, padronize markdown entre candidatos, trate respostas longas como sinal de revisão (alinha com o cap de ~600 chars/~6 bolhas). ([MT-Bench, arXiv 2306.05685](https://arxiv.org/html/2306.05685v4); [Judging the Judges, arXiv 2604.23178](https://arxiv.org/html/2604.23178)) ⚠️ **contestado:** as duas leituras (verbosidade forte histórica vs. estilo dominante hoje) coexistem e estão ambas verificadas; o reframe de 2026 é o estado atual.

**[alto] Espelhe o padrão τ-bench: cada fixture checa estado final do banco E adesão a políticas de domínio, incluindo abstenção.** τ-bench/τ²-bench verificam mudança no banco + interação com usuário simulado + adesão a políticas (τ²-bench adiciona dual-control); BFCL v4 mede capacidade de abstenção em cenários stateful. Teste Piso de desconto, isolamento por par, Pix nunca trava, e casos de escalar/recusar — não só caminhos felizes. ([τ-bench, arXiv 2406.12045](https://arxiv.org/abs/2406.12045); [τ²-Bench, arXiv 2506.07982](https://arxiv.org/abs/2506.07982); [BFCL v3 Multi-Turn](https://gorilla.cs.berkeley.edu/blogs/13_bfcl_v3_multi_turn.html))

**[médio] Modele graders LLM como decisões binárias por critério isolado, não nota 1-5 agregada.** Binário força rótulo consistente; desenvolver um juiz exige 100+ exemplos rotulados e manutenção contínua; grade cada dimensão com juiz isolado e dê opção "Unknown". Ex.: "vazou que é IA? sim/não", "recusou fetiche não oferecido? sim/não". ([Hamel Husain — LLM Evals FAQ](https://hamel.dev/blog/posts/evals-faq/); [Anthropic — Demystifying evals](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents))

**[médio] Não use Sonnet 4.6 como agente E único juiz comparativo sem controle.** Self-preference é instável entre modelos (Gemini Flash +0,56 a favor de si; GPT-4o -0,48); mitigações são modelo-dependentes (swap sozinho deu só +0,2pp p/ Sonnet 4). Quando inevitável, ancore com referência/rubrica e valide contra rótulos humanos; re-meça ao migrar de versão. ([Judging the Judges, arXiv 2604.23178](https://arxiv.org/html/2604.23178)) ⚠️ **contestado:** evidência de UMA fonte primária (2604.23178) sem replicação independente das cifras de self-preference; sobrevive por ser fonte robusta, mas a recomendação é prudencial.

## Observabilidade e custo

**[crítico] Leia `usage` por turno e emita métrica de cache hit rate; ambos `cache_*` em 0 = nada cacheado, sem erro.** Total = `cache_read_input_tokens` (0,1x) + `cache_creation_input_tokens` (1,25x 5m / 2,0x 1h) + `input_tokens` (após o último breakpoint, preço cheio); hit rate = `cache_read / total`. No nó llm, leia `usage_metadata['input_token_details']['cache_read']`/`['cache_creation']`; alerte quando o hit rate cair abaixo de ~40-50% no chat (prefixo geral deveria estar quente → breakpoint mal colocado ou prefixo <1024 tokens). ([Anthropic — Prompt caching](https://platform.claude.com/docs/en/build-with-claude/prompt-caching); [anthropic-sdk-python #1194](https://github.com/anthropics/anthropic-sdk-python/issues/1194)) ⚠️ ressalva histórica: na prática o Sonnet já pareceu começar a cachear só a partir de ~2048 tokens — trate 1024 como teto de risco e meça empiricamente.

**[crítico] Ligue tracing LangSmith no piloto com anonymizer de PII OBRIGATÓRIO antes de 100%.** Tracing é ativado por env vars (`LANGSMITH_TRACING`/`API_KEY`/`PROJECT`), flui assíncrono sem latência, cada nó vira um run. O conteúdo da conversa é íntimo e tem PII sensível (nomes, endereços): configure `create_anonymizer` cobrindo o PAYLOAD inteiro, não só o SSN do exemplo; migre para sampling via `tracing_context` quando o volume crescer. ([LangChain — LangGraph Observability](https://docs.langchain.com/oss/python/langgraph/observability); [Mask inputs/outputs](https://docs.langchain.com/langsmith/mask-inputs-outputs))

**[alto] Avalie TTL 1h no breakpoint do prefixo geral; monitore `ephemeral_5m` vs `ephemeral_1h`.** Read 0,1x, write 1,25x (5m) / 2,0x (1h); o cache 5m só renova sem custo enquanto há hits dentro da janela — gap > TTL força re-write. Meça antes/depois: se a maioria dos turnos chega com gap >5min e você paga re-write, o 2,0x do 1h sai muito mais barato que perder o cache. ([Anthropic — Prompt caching](https://platform.claude.com/docs/en/build-with-claude/prompt-caching))

**[alto] Cadastre o preço de `cache_read` do Sonnet 4.6 (0,1x) e os tipos multimodais na tabela de pricing do LangSmith.** O LangSmith deriva custo de `usage_metadata` com cálculo greedy (preço de `cache_read` aos cacheados, default ao resto). Sem cadastrar, o dashboard infla o custo (cobra cache como input cheio). Garanta que o nó llm propague `input_token_details.cache_read`. ([LangChain — Cost tracking](https://docs.langchain.com/langsmith/cost-tracking); [InputCostDetails schema](https://reference.langchain.com/python/langsmith/schemas/InputCostDetails)) ⚠️ ressalva: existe bug conhecido de dupla contagem (`cache_creation` vs `ephemeral_5m`) — exige SDK sem o bug. ([langsmith-sdk #2150](https://github.com/langchain-ai/langsmith-sdk/issues/2150))

**[alto] Separe p95 de TTFT (percepção) do p95 de duração total (custo de tempo).** A Anthropic distingue baseline latency de time-to-first-token; streaming melhora a responsividade percebida sem reduzir o total. Como a humanização envia em bolhas pós-resposta-completa (não streama ao cliente), o p95 que importa para custo de turno é a duração total da chamada; TTFT detecta degradação de upstream da Anthropic separadamente. ([Anthropic — Reducing latency](https://platform.claude.com/docs/en/test-and-evaluate/strengthen-guardrails/reduce-latency))

**[alto] Configure metric alert de custo/tokens por janela no Sentry como defesa contra spike.** O Sentry rastreia tokens/custo por modelo/endpoint, dispara alerta quando custo/tokens cruzam limiar e em spikes súbitos (cliente adversário, loop de tool, injeção que dispara mídia em massa). Separe o spike de custo do spike de escaladas por exaustão/refusal. ([Sentry Blog — AI agent observability](https://blog.sentry.io/ai-agent-observability-developers-guide-to-agent-monitoring/); [Sentry — Monitor AI agent costs](https://sentry.io/cookbook/monitor-ai-agent-costs-nextjs/))

**[alto] Monte um dashboard com input/cached/hit-rate por workload (chat, vision-Pix) e vigie o anti-padrão de não-determinismo.** As 3 causas de hit rate 0% são: prefixo abaixo do mínimo, valor não-determinístico no texto cacheado (timestamp/ID/ordenação de dict), ou mensagens reembaralhadas — já um bug conhecido do projeto (turno_id/score no prefixo zerava o cache). O prefixo geral deve estabilizar bem acima de 60%. ([Anthropic — Prompt caching](https://platform.claude.com/docs/en/build-with-claude/prompt-caching); [Start Debugging](https://startdebugging.net/2026/04/how-to-add-prompt-caching-to-an-anthropic-sdk-app-and-measure-the-hit-rate/)) ⚠️ **parcial:** os thresholds "~60% / 50% / 1 semana" são heurística de blog de terceiros, não spec da Anthropic — calibre com a baseline real do workload.

## Resiliência

**[crítico] Inspecione `response.stop_reason` ANTES de tratar o conteúdo — `try/except` do SDK é cego a ele.** `stop_reason` chega SEMPRE em 200 OK no corpo (a doc separa "Stop reasons (successful responses)" de "Errors HTTP 4xx/5xx"). `refusal`/`max_tokens`/`model_context_window_exceeded`/`end_turn` vazio NÃO disparam `APIStatusError`. Trate-os como ramos explícitos (`Command/goto`), nunca confiando no retry de exceção. ([Anthropic Docs — Handling stop reasons](https://platform.claude.com/docs/en/api/handling-stop-reasons); [Python SDK](https://platform.claude.com/docs/en/api/sdks/python))

**[alto] Trate `refusal` como escalada determinística para Fernando/handoff (`modelo_recusou`), nunca reenvie cru ao cliente.** `refusal` é stop_reason em 200 OK; em Opus 4.7+ vem com `stop_details.category` (`cyber`/`bio`/null) — NÃO faça parsing de `explanation` (não é estável). Filtros ASL-3/CBRN geram falsos positivos; como o agente conversa sobre conteúdo adulto/íntimo, monitore a taxa de refusal e logue `category` separadamente para detectar falso-positivo recorrente. ([Anthropic — Handling stop reasons](https://platform.claude.com/docs/en/api/handling-stop-reasons); [Anthropic Support — Sonnet 4.5 safety filters](https://support.claude.com/en/articles/12449294-understanding-sonnet-4-5-s-api-safety-filters))

**[alto] NÃO escreva camada manual de retry/backoff por cima do SDK para 429/5xx/timeout.** O SDK retenta automaticamente 2x com backoff exponencial (conexão, 408, 409, 429, ≥500), respeita `Retry-After` e `x-should-retry`; timeout padrão 10 min. Retry manual empilha tentativas e causa hangs de 10+ min. Delegue ao SDK e só ajuste `max_retries`/`timeout`; reserve lógica própria para `stop_reason`. ([Anthropic — Python SDK](https://platform.claude.com/docs/en/api/sdks/python); [_base_client.py](https://github.com/anthropics/anthropic-sdk-python/blob/main/src/anthropic/_base_client.py))

**[alto] Distinga `max_tokens` (saída truncada) de `model_context_window_exceeded` (input grande demais).** O segundo (stop_reason em Sonnet 4.5+, sem beta header) significa que a resposta bateu na janela antes do `max_tokens` — não adianta aumentar `max_tokens`, precisa podar histórico/resumir antes de reenviar. Logue como sinal de que prompt-cache+histórico cresceu além do orçamento. ([Anthropic — Handling stop reasons](https://platform.claude.com/docs/en/api/handling-stop-reasons))

**[alto] Não duplique backoff: o SDK já retenta 429/5xx; ponha `RetryPolicy` do LangGraph para falhas do próprio nó.** `RetryPolicy` por nó (defaults: max_attempts=3, backoff exponencial; para HTTP só retenta 5xx, não 4xx) cobre LLM, tools de domínio, STT, vision — mas há sobreposição na camada de erro de API. Escolha uma camada para o erro de API. **Crítico para o projeto:** sem checkpointer não há restart do último superstep nem pending writes persistidos — o retry fica só em memória por execução. ([LangChain — RetryPolicy](https://reference.langchain.com/python/langgraph/types/RetryPolicy); [Durable execution](https://docs.langchain.com/oss/python/langgraph/durable-execution); [Changelog](https://changelog.langchain.com/announcements/enhanced-state-management-retries-in-langgraph-python))

**[alto] Torne efeitos colaterais idempotentes por chave determinística; use streaming em requests longos.** Em timeout não-streaming o cliente termina a conexão e retenta SEM ter recebido resposta — um turno pode ser reenviado após o servidor processá-lo, e o Messages API não expõe `Idempotency-Key`. Mantenha `turno_id` uuid5, dedupe de card por `card_message_id`, advisory lock + EXCLUDE na agenda, `registrar_envio` só após sucesso. Nunca dispare envio ao cliente/escrita em banco no caminho re-executável sem guarda. ([Anthropic — Python SDK / Long requests](https://platform.claude.com/docs/en/api/sdks/python))

**[médio] NÃO envolva chamadas com retry genérico que pegue 4xx — falhe rápido em 400/401/403/422.** O SDK não retenta 4xx não-transitórios por design (são bug de código/credencial/payload). Escale-os como erro de infra (alerta Sentry), não como degradação transitória — espelha a distinção recuperável (string ao modelo) vs infra (exceção). ([Anthropic — Python SDK](https://platform.claude.com/docs/en/api/sdks/python))

**[médio] Se `stop_reason=='max_tokens'` e o último bloco for `tool_use`, reexecute com `max_tokens` maior antes de despachar.** Senão a tool call sai corrompida. Para texto truncado ao cliente, prefira métrica+log (`TURNO_TRUNCADO`) a continuação automática (bolhas de WhatsApp devem ser curtas). ([Anthropic — Handling stop reasons](https://platform.claude.com/docs/en/api/handling-stop-reasons))

**[médio] Monte o turno user com APENAS o `tool_result` (sem bloco de texto extra após).** `end_turn` pode vir vazio (2-3 tokens) após `tool_result` porque texto após tool_result "ensina" o modelo a encerrar; reenviar a resposta vazia não resolve ("Claude already decided it's done"). Detecte AIMessage vazio com `stop_reason=end_turn` e trate como "turno sem resposta" (bug-irmão já visto, commit 4a45993), não envie bolha vazia. ([Anthropic — Handling stop reasons](https://platform.claude.com/docs/en/api/handling-stop-reasons))

**[baixo] Trate `pause_turn` como caso inerte — não construa loop de continuação.** `pause_turn` só ocorre com SERVER TOOLS (web_search/web_fetch); o agente usa tools client-side. O esgotamento de iterações do agente é controlado por `recursion_limit` do LangGraph (capturar `GraphRecursionError` → escalar), não por `pause_turn`. ([Anthropic — Handling stop reasons](https://platform.claude.com/docs/en/api/handling-stop-reasons))

**[alto] ⚠️ contestado — fallback de modelo por refusal: MANTENHA refusal→escalada a humano, não fallback silencioso.** A Anthropic sugere, em documentos separados, tentar Haiku 4.5 (Tip em stop-reasons) ou retentar com Sonnet 4 (artigo de suporte) para contornar filtro de segurança. Para o domínio adulto da Elite Baby, fallback automático pode mascarar conteúdo que deveria escalar; o projeto já removeu o fallback Haiku. Só considere troca de modelo se métricas mostrarem falso-positivo sistêmico em conteúdo legítimo. ([Anthropic — Handling stop reasons](https://platform.claude.com/docs/en/api/handling-stop-reasons); [Anthropic Support — safety filters](https://support.claude.com/en/articles/12449294-understanding-sonnet-4-5-s-api-safety-filters)) ⚠️ a claim funde duas recomendações distintas (Haiku 4.5 vs Sonnet 4, modelos diferentes em docs diferentes) e há drift de versão (o projeto roda Sonnet 4.6, as Tips citam 4.5/4.1/4.5/4).

## Multimodalidade

**[crítico] Trate extração de valor/chave/status do Pix como NÃO confiável; não recomprima JPEG; rejeite <200px.** A Anthropic documenta que o modelo alucina em imagens de baixa qualidade/rotacionadas/muito pequenas (<200px), e compressão lossy agressiva (especialmente múltiplos passes) prejudica leitura de texto. Mantenha a decisão de produto: Pix nunca trava; divergência/suspeita → duvidoso → fila assíncrona de Fernando; envie o que veio do MinIO apenas redimensionado, sem segundo passe de compressão. ([Anthropic — Vision: Limitations / Ensure image quality](https://platform.claude.com/docs/en/build-with-claude/vision))

**[alto] Redimensione para ≤1568px na borda longa (≈1,15 MP) ANTES de enviar ao Sonnet 4.6 e antes do base64.** Limite 5MB/imagem (10MB claude.ai), dim. máx 8000x8000 (rejeita acima); custo ~largura×altura/750. O Sonnet 4.6 redimensionaria para 1568px de qualquer forma — fazer antes corta latência de TTFT, custo e evita 5xx. Valide o tamanho no worker logo após o download do MinIO. ([Anthropic — Vision](https://platform.claude.com/docs/en/build-with-claude/vision)) ⚠️ **parcial:** o cap de 1568px é função do MODELO, não do contexto de 200k — Opus 4.7/4.8 usam 2576px; a conclusão prática (1568px p/ Sonnet 4.6) está correta, a justificativa "porque é 200k" é falsa.

**[alto] Use Strict Tool Use (`strict:true`) ou `json_schema` na extração do Pix; fixe o schema; valide valor mínimo em Python.** Structured Outputs (GA em Sonnet 4.6) garante conformidade sem retries; a gramática é compilada na 1ª request e cacheada 24h — reutilize o schema (não gere dinâmico). Trocar `output_config.format` invalida o prompt cache. Schemas exigem `additionalProperties:false` e NÃO suportam `minimum`/`maximum` — valide o valor mínimo do Pix no worker, não no schema. ([Anthropic — Structured Outputs](https://platform.claude.com/docs/en/build-with-claude/structured-outputs)) ⚠️ ressalva menor: a latência exata "~500-1000ms" não consta verbatim na doc (diz só "additional latency while grammar compiles").

**[alto] Verifique `stop_reason` ANTES de confiar no `parsed_output` do vision; defina `max_tokens` com folga.** Structured Outputs pode NÃO conformar quando há `refusal` (a recusa precede o schema) ou `max_tokens` (JSON incompleto) — ambos chegam em 200 OK e furam tratamento por exceção. Se for `refusal`/`max_tokens`, marque o comprovante duvidoso → fila de Fernando, em vez de assumir JSON válido; use `max_tokens` ~2048 para o schema do comprovante. ([Anthropic — Structured Outputs: Refusals e max_tokens](https://platform.claude.com/docs/en/build-with-claude/structured-outputs))

**[alto] Monte o content block com a imagem ANTES do texto e NUNCA persista base64 no histórico do checkpoint.** "Claude works best when images come before text"; em multi-turno base64 é reenviado a cada turno, inflando payload/latência. Faça a extração em chamada isolada/efêmera (ou Files API com `file_id`) e grave só o resultado estruturado no estado — protege o prompt caching de invalidação e reduz custo por turno. ([Anthropic — Vision: Prompt examples / Files API](https://platform.claude.com/docs/en/build-with-claude/vision))

**[alto] Considere Groq whisper-large-v3-turbo como STT primário pelo custo/latência, com gpt-4o-transcribe como fallback de acurácia.** Groq turbo a ~US$0,04/h (~9x mais barato que OpenAI), ~216x tempo real, endpoint OpenAI-compatível (adapter trocável por config); gpt-4o-transcribe tem WER menor (~4,1% vs ~5,3%) e streaming. Sempre passe `language='pt'`; converta para WAV/FLAC 16kHz mono antes de enviar (o modelo faz downsample de qualquer forma); teto de 25MB (free) exige chunking. ([Groq — Speech to Text](https://console.groq.com/docs/speech-to-text); [Groq Blog — Whisper Large v3 Turbo](https://groq.com/blog/whisper-large-v3-turbo-now-available-on-groq-combining-speed-quality-for-speech-recognition); [TokenMix](https://tokenmix.ai/blog/gpt-4o-transcribe-vs-whisper-review-2026); [OpenAI — gpt-4o-transcribe](https://developers.openai.com/api/docs/models/gpt-4o-transcribe)) ⚠️ **parcial:** WER (4,1%/5,3%) e janela de streaming (500-1500ms) vêm de blog terceiro, não da página oficial OpenAI; o "~189x" do whisper-large-v3 na Groq não confere (oficiais citam 164x / 299x). O parâmetro `prompt` (~224-244 tokens) e os limites/preços são confirmados.

**[alto] Aplique VAD (ex.: SileroVAD) para recortar silêncio/ruído antes do STT; trate transcrição curta/suspeita com cautela.** Whisper alucina texto fantasma em silêncios e não-fala (~40% das inferências em não-fala; 52,1% em clips de 1s); VAD reduz drasticamente o WER. Áudios de WhatsApp têm respiração/ruído/pausas; transcrição vazia/ruidosa deve degradar para placeholder em vez de alimentar conteúdo alucinado no turno. ([arXiv 2501.11378](https://arxiv.org/abs/2501.11378); [Calm-Whisper, arXiv 2505.12969](https://arxiv.org/abs/2505.12969)) ⚠️ **parcial:** o número "WER 0,675→0,419" foi atribuído à fonte errada (é do WhisperAlign, [arXiv 2603.04809](https://arxiv.org/html/2603.04809v1), não do 2501.11378); e "gpt-4o-transcribe herda parte do comportamento" é inferência não corroborada por fonte primária. O núcleo (VAD reduz alucinação) é sólido.

**[médio] Persista a mídia original no MinIO por `object_key` — a Anthropic deleta a imagem após processar.** Imagens via API são efêmeras (não armazenadas, não usadas para treino) e o modelo não lê metadata. NÃO confie em re-fetch pela API: o card "cliente chegou"/"saída confirmada" e a auditoria apontam para o `object_key`; para o vision, leia do MinIO, redimensione e envie em chamada efêmera sem persistir base64 no checkpoint. ([Anthropic — Vision FAQ](https://platform.claude.com/docs/en/build-with-claude/vision))

**[médio] Para imagem referenciada em mais de uma chamada, use Files API (`file_id`) com header `files-api-2025-04-14`.** Em multi-turno, base64 reenviado cresce payload/latência conforme o histórico acumula; `file_id` mantém o payload pequeno. Para extração única do Pix, uma chamada efêmera inline basta — o crítico é nunca deixar os bytes entrarem no histórico persistido do LangGraph. ([Anthropic — Vision: Files API](https://platform.claude.com/docs/en/build-with-claude/vision))

## Lacunas e incertezas

**Itens contestados (incluídos com ressalva):**
- **Modelos maiores drifam mais** ([arXiv:2412.00804](https://arxiv.org/abs/2412.00804)): confirmado NA fonte, mas é UM estudo sem replicação independente em larga escala; a generalização "modelo forte não protege a voz" depende deste corpo restrito.
- **Viés de verbosidade vs. estilo** ([MT-Bench](https://arxiv.org/html/2306.05685v4) vs [arXiv 2604.23178](https://arxiv.org/html/2604.23178)): duas leituras coexistem (verbosidade forte histórica 2023 vs. estilo/markdown dominante e preferência por concisão em 2026); ambas verificadas.
- **JailbreakBench / robustez do Claude** ([arXiv 2404.01318](https://arxiv.org/abs/2404.01318), [2404.02151](https://arxiv.org/html/2404.02151v3)): os 100% via prefilling são Claude 2.x–3.5, não Opus 4.5/4.7 — ressalva de versão.
- **Self-preference do juiz** ([arXiv 2604.23178](https://arxiv.org/html/2604.23178)): cifras de UMA única fonte primária, sem replicação independente.
- **Fallback de modelo por refusal** ([stop reasons](https://platform.claude.com/docs/en/api/handling-stop-reasons) + [safety filters](https://support.claude.com/en/articles/12449294-understanding-sonnet-4-5-s-api-safety-filters)): claim funde duas recomendações de docs separados (Haiku 4.5 vs Sonnet 4) + drift de versão (projeto em Sonnet 4.6).
- **Regressão de TTL 1h→5min** ([claude-code #46829](https://github.com/anthropics/claude-code/issues/46829)): relato real e detalhado, mas issue fechada como "not planned" SEM confirmação oficial; só a existência do relato sobrevive, a regressão em si é não-confirmada.

**Itens com atribuição imprecisa (recomendação válida, fonte/número a corrigir):**
- "Lost-in-the-middle" não é afirmação DA Anthropic no artigo de context engineering (é Liu et al. 2023).
- "trim_messages por TOKENS, não mensagens" descreve uso típico, não restrição da API.
- BFCL "2.000+ pares" e "~0,73 teto" não conferem (topo V4 atual ~0,72-0,77, varia por versão).
- "405 pesquisadores" dos Constitutional Classifiers e "~1% compute" (é da versão ++, não da original).
- Limiares de persona "~50/100+ turnos" e thresholds de cache hit rate "60%/50%/1 semana" são heurística, não spec.
- WER do Whisper "~7,4%" só vale em benchmark limpo; "189x" da Groq e WER/streaming do gpt-4o-transcribe repousam em fontes secundárias.
- Número "WER 0,675→0,419" pertence ao WhisperAlign (arXiv 2603.04809), não ao 2501.11378.

**Descartados por falta de corroboração (não listados acima como recomendações):** afirmações cujo `verdict` foi `sem_corroposição`/`sem_corroboracao` no dataset — alguns verdicts vieram desalinhados de seus claims no JSON de entrada (ex.: pares pre-warming/automatic-caching, multimodalidade smoke-test de vision, resiliência de degradação graciosa de mídia, "how we contain Claude" como claim isolado), onde o objeto `verdict` não correspondia ao `claim`. Onde o conteúdo do claim era de fato corroborado por outro verdict do mesmo eixo (ex.: pre-warming confirmado em verdict adjacente; degradação graciosa de mídia sustentada por RetryPolicy + stop-reasons já confirmados), a recomendação foi mantida; onde não havia verdict válido correspondente, foi omitida.
